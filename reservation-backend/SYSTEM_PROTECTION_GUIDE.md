# 系統保護機制指南

## 🛡️ 概述

本系統已實作多層次的保護機制，以防止學生預約行為導致系統卡死或無法運作。這些保護機制涵蓋了從前端到後端的各個層面。

## 🚨 識別的主要風險

### 1. 資料庫連接池耗盡
- **風險**：大量併發請求導致資料庫連接池耗盡
- **影響**：系統完全無法響應新請求
- **保護措施**：
  - 增加連接池大小（max: 50, min: 10）
  - 連接池監控和警告機制
  - 請求超時控制

### 2. 重複預約攻擊
- **風險**：學生快速點擊或使用腳本重複提交
- **影響**：資料庫鎖定、事務衝突
- **保護措施**：
  - 前端提交頻率限制（5秒內最多3次）
  - 後端頻率限制（每分鐘最多3次預約）
  - 併發控制鎖機制

### 3. 大量併發預約
- **風險**：熱門活動開放時大量學生同時預約
- **影響**：資料庫死鎖、系統響應緩慢
- **保護措施**：
  - 活動級別的併發控制（每活動最多5個併發請求）
  - 容量預檢查機制
  - 佇列式處理

### 4. 惡意輸入攻擊
- **風險**：超長字串、特殊字符、SQL注入
- **影響**：資料庫錯誤、記憶體溢出
- **保護措施**：
  - 輸入長度限制
  - SQL注入防護
  - 輸入清理和驗證

### 5. 時間窗口競爭
- **風險**：預約時間開放瞬間的競爭條件
- **影響**：名額計算錯誤、重複預約
- **保護措施**：
  - 資料庫行鎖定
  - 原子性事務處理
  - 容量狀態快照

## 🔧 實作的保護機制

### 1. 頻率限制（Rate Limiting）

#### 預約請求限制
```javascript
// 每分鐘最多 3 次預約請求
reservationLimiter: {
  windowMs: 60 * 1000,
  max: 3,
  keyGenerator: (req) => `reservation:${ip}:${studentId}`
}
```

#### 一般 API 限制
```javascript
// 每分鐘最多 60 次一般請求
generalLimiter: {
  windowMs: 60 * 1000,
  max: 60
}
```

#### 登入限制
```javascript
// 每分鐘最多 5 次登入嘗試
loginLimiter: {
  windowMs: 60 * 1000,
  max: 5
}
```

### 2. 併發控制（Concurrency Control）

#### 活動級別併發限制
- 每個活動最多同時處理 5 個預約請求
- 防止同一學生重複請求
- 自動清理過期請求（30秒超時）

#### 容量預檢查
- 在事務開始前檢查活動容量
- 避免不必要的資料庫鎖定
- 提供即時容量資訊

### 3. 輸入驗證和清理

#### 長度限制
- 學號：最大 20 字符
- 姓名：最大 100 字符
- Email：最大 255 字符
- 請求體：最大 10MB

#### 格式驗證
- 學號格式：B/M/D/I/J + 9位數字
- 姓名格式：中文或英文
- Email 格式：標準 email 格式

#### SQL 注入防護
- 危險字符檢測
- 輸入清理和轉義
- 參數化查詢

### 4. 資料庫優化

#### 連接池配置
```javascript
pool: {
  max: 50,          // 最大連接數
  min: 10,          // 最小連接數
  acquire: 30000,   // 獲取連接超時
  idle: 10000,      // 連接空閒時間
  evict: 1000,      // 清理間隔
  handleDisconnects: true
}
```

#### 索引優化
- 預約表複合索引：`(eventId, studentId)`, `(eventId, studentEmail)`
- 用戶表索引：`studentId`, `email`, `isBlacklisted`
- 活動表索引：`date`, `eventType`, `(date, startTime)`

### 5. 錯誤處理和監控

#### 全域錯誤處理
- 統一錯誤格式
- 詳細錯誤日誌
- 開發/生產環境區分

#### 系統監控
- 健康檢查端點：`/api/health`
- 系統狀態端點：`/api/status`
- 連接池監控
- 記憶體使用監控

## 🚀 部署建議

### 1. 環境變數配置
```bash
# 資料庫配置
DB_HOST=localhost
DB_USER=root
DB_PASSWORD=your_password
DB_NAME=activity_reservation

# Redis 配置（可選，用於分散式頻率限制）
REDIS_URL=redis://localhost:6379

# 環境設定
NODE_ENV=production
```

### 2. 伺服器配置
- 建議使用 PM2 進行進程管理
- 配置日誌輪轉
- 設置監控和警報

### 3. 負載平衡
- 使用 Nginx 進行負載平衡
- 配置會話粘性（如果使用記憶體存儲）
- 考慮使用 Redis 進行分散式存儲

## 🔍 故障排除

### 常見問題

#### 1. 頻率限制過於嚴格
- 檢查 `rateLimiter.js` 中的配置
- 調整 `max` 和 `windowMs` 參數
- 考慮使用 Redis 進行分散式限制

#### 2. 併發控制阻塞正常請求
- 檢查 `concurrencyControl.js` 中的 `maxConcurrentPerEvent`
- 調整 `requestTimeout` 參數
- 使用 `/api/reset-concurrency` 端點重置狀態

#### 3. 資料庫連接池耗盡
- 增加 `pool.max` 配置
- 檢查長時間運行的查詢
- 優化資料庫索引

#### 4. 記憶體使用過高
- 檢查記憶體洩漏
- 調整 Node.js 記憶體限制
- 考慮使用集群模式

### 監控命令
```bash
# 檢查系統健康狀態
curl http://localhost:3000/api/health

# 檢查詳細系統狀態
curl http://localhost:3000/api/status

# 重置併發控制（僅開發環境）
curl -X POST http://localhost:3000/api/reset-concurrency
```
