# 培力英檢報名失敗問題分析

## 問題描述
報名 10 筆資料僅成功 1 筆，失敗率 90%

## 可能原因分析

### 1. **學號唯一性約束衝突** ⚠️ 高機率
- **問題**：`studentId` 欄位設有 `unique: true` 約束
- **原因**：
  - 用戶重複提交（快速點擊、網路延遲導致重複請求）
  - 檢查和建立之間存在競爭條件（Race Condition）
  - 第一次檢查時沒有記錄，但在建立時已有記錄
- **影響**：會拋出 `SequelizeUniqueConstraintError`
- **解決方案**：
  - 使用資料庫事務（Transaction）確保原子性
  - 使用 `findOrCreate` 或 `upsert` 方法
  - 捕獲唯一約束錯誤並返回友好訊息

### 2. **必填欄位驗證失敗** ⚠️ 高機率
- **問題**：模型定義中多個欄位為 `allowNull: false`
- **必填欄位列表**：
  - `email` - 電子郵件
  - `studentNameZh` - 中文姓名
  - `lastNameEn` - 英文拼音姓
  - `firstNameEn` - 英文拼音名
  - `birthDate` - 出生年月日
  - `phone` - 行動電話
  - `postalCode` - 郵遞區號
  - `city` - 縣市
  - `district` - 行政區
  - `address` - 詳細地址
  - `degreeLevel` - 就讀身分
  - `grade` - 年級
  - `college` - 學院
  - `department` - 科系
  - `idPhoto` - 證件照（檔案路徑）
  - `agreedToTerms` - 同意事項（必須為 true）
  - `infoSource` - 資訊來源
- **原因**：
  - 前端表單驗證不完整，允許空值提交
  - FormData 中某些欄位為空字串或 undefined
  - 條件式必填欄位（如 B2 相關欄位）處理不當
- **影響**：會拋出 `SequelizeValidationError`
- **解決方案**：
  - 加強前端驗證
  - 後端提供預設值（但需符合業務邏輯）
  - 明確區分可選和必填欄位

### 3. **檔案上傳問題** ⚠️ 中機率
- **問題**：`idPhoto` 為必填，但檔案可能上傳失敗
- **原因**：
  - 檔案大小超過 5MB 限制
  - 檔案格式不符合（只允許 JPG、PNG、PDF）
  - 檔案上傳過程中網路中斷
  - Multer 錯誤處理不當
- **影響**：檔案上傳失敗導致整個請求失敗
- **解決方案**：
  - 檢查 Multer 錯誤處理中間件是否正確放置
  - 提供更詳細的檔案上傳錯誤訊息
  - 考慮將檔案上傳改為可選（如果業務允許）

### 4. **JSON 欄位解析失敗** ⚠️ 中機率
- **問題**：`passedExamTypes`、`disabilityTypes`、`examAssistanceOptions` 需要 JSON 解析
- **原因**：
  - 前端傳送的 JSON 字串格式錯誤
  - 空字串或 undefined 導致解析失敗
  - 雖然有 try-catch，但可能影響後續邏輯
- **影響**：陣列欄位可能為空，但不應導致整體失敗
- **解決方案**：
  - 已實作安全解析，但需確認所有情況都處理到

### 5. **資料類型不匹配** ⚠️ 中機率
- **問題**：某些欄位的資料類型可能不匹配
- **原因**：
  - `birthDate` 需要 DATEONLY 格式，但前端可能傳送錯誤格式
  - `agreedToTerms` 需要 BOOLEAN，但 FormData 傳送的是字串
  - 字串長度超過限制（如 `address` 限制 200 字元）
- **影響**：Sequelize 驗證失敗
- **解決方案**：
  - 加強資料類型轉換
  - 驗證字串長度

### 6. **並發競爭條件** ⚠️ 中機率
- **問題**：多個請求同時檢查和建立記錄
- **原因**：
  - 用戶快速點擊提交按鈕
  - 網路延遲導致重複請求
  - 檢查 `existing` 和 `create` 之間沒有鎖定
- **影響**：多個請求同時通過檢查，但只有一個能成功建立
- **解決方案**：
  - 使用資料庫事務
  - 使用 `findOrCreate` 方法
  - 前端防重複提交機制

### 7. **錯誤處理不當** ⚠️ 低機率
- **問題**：錯誤被吞掉或沒有正確回傳
- **原因**：
  - catch 區塊中的錯誤處理不完整
  - 錯誤訊息沒有正確傳遞給前端
  - 某些錯誤類型沒有被正確捕獲
- **影響**：前端收到成功回應，但實際上沒有建立記錄
- **解決方案**：
  - 檢查所有錯誤處理路徑
  - 確保錯誤訊息正確回傳

### 8. **資料庫連接問題** ⚠️ 低機率
- **問題**：資料庫連接池耗盡或超時
- **原因**：
  - 大量並發請求
  - 資料庫連接未正確釋放
  - 連接池設定過小
- **影響**：請求超時或連接失敗
- **解決方案**：
  - 檢查資料庫連接池設定
  - 確保連接正確釋放

## 建議的修正方案

### 優先級 1：立即修正
1. **加入資料庫事務**：確保檢查和建立的原子性
2. **改進錯誤處理**：捕獲所有可能的錯誤類型
3. **加強資料驗證**：在建立前驗證所有必填欄位

### 優先級 2：短期改進
1. **使用 findOrCreate**：避免競爭條件
2. **改進前端驗證**：確保所有必填欄位都有值
3. **檔案上傳改進**：提供更詳細的錯誤訊息

### 優先級 3：長期優化
1. **加入重試機制**：對於暫時性錯誤
2. **加入日誌記錄**：記錄所有失敗的報名嘗試
3. **監控和告警**：監控報名成功率

## 除錯建議

1. **檢查後端日誌**：
   - 查看是否有 `SequelizeValidationError`
   - 查看是否有 `SequelizeUniqueConstraintError`
   - 查看是否有其他錯誤訊息

2. **檢查資料庫**：
   - 確認 `english_test_registrations` 表結構正確
   - 檢查是否有重複的學號記錄
   - 查看失敗的記錄（如果有部分資料寫入）

3. **測試單筆報名**：
   - 使用 Postman 或 curl 測試單筆報名
   - 確認所有必填欄位都有值
   - 確認檔案上傳正常

4. **檢查前端表單**：
   - 確認所有必填欄位都有驗證
   - 確認表單提交前所有欄位都有值
   - 檢查瀏覽器 Console 是否有錯誤
