# 系統優化記錄

## 2024-12-11：資料庫連接池優化（第二次調整）

### 變更內容

**檔案**：`reservation-backend/db.js`

**變更前配置**：
```javascript
pool: {
  max: 5,    // 最大連接數：5
  min: 0,    // 最小連接數：0
  acquire: 30000,
  idle: 10000
}
```

**第一次優化後配置**：
```javascript
pool: {
  max: 50,          // 最大連接數：50
  min: 10,          // 最小連接數：10
  acquire: 30000,
  idle: 10000,
  evict: 1000,
  handleDisconnects: true
}
```

**最終配置**（根據伺服器規格和預期負載調整）：
```javascript
pool: {
  max: 120,         // 最大連接數：120（支援 250 個並發學生）
  min: 20,          // 最小連接數：20（保持基本連接池）
  acquire: 30000,   // 獲取連接的最大等待時間（30秒）
  idle: 10000,      // 連接空閒時間（10秒後釋放）
  evict: 1000,      // 檢查空閒連接的間隔（1秒）
  handleDisconnects: true  // 自動處理斷線重連
}
```

### 伺服器規格

- **CPU**: Intel I5 7500 (4核心4線程)
- **RAM**: 16GB
- **預期並發負載**: 250個學生同時進行預約

### MySQL 配置建議

**MySQL `max_connections`**: **350**

詳細配置說明請參考：`MYSQL_CONFIG_RECOMMENDATION.md`

### 優化原因

根據《報名系統並發性能評估報告》，原配置存在以下問題：
1. **連接池過小**：只能同時處理 5 個資料庫連接，無法應對大量並發報名
2. **缺少最小連接數**：`min: 0` 導致每次請求都需要建立新連接，增加延遲
3. **缺少連接管理機制**：沒有 `evict` 和 `handleDisconnects` 配置

### 預期效果

- **並發處理能力**：從約 5-10 個同時請求提升到 50-100 個
- **響應時間**：減少連接建立時間，提高響應速度
- **系統穩定性**：自動處理斷線重連，提高系統可靠性
- **資源利用**：保持最小連接池，減少連接建立開銷

### 注意事項

1. **資料庫伺服器配置**：確保 MySQL 伺服器的 `max_connections` 設定足夠大（建議至少 100+）
2. **記憶體使用**：連接池增大會增加記憶體使用，需要監控伺服器資源
3. **重啟服務**：變更後需要重啟後端服務才能生效

### 驗證步驟

1. 重啟後端服務
2. 檢查服務日誌，確認資料庫連接成功
3. 監控連接池使用情況（可通過 Sequelize 的連接池監控功能）
4. 進行負載測試，驗證並發處理能力
