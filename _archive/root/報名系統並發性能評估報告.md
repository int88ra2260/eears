# 報名系統並發性能評估報告

## 📊 執行摘要

本報告評估了培力英檢報名系統在短時間內接受大量學生同時報名的能力。經過代碼審查，發現系統在並發處理方面存在多個潛在瓶頸，需要進行優化以確保在高負載情況下穩定運行。

**評估結論**：系統目前**無法有效應對大量並發報名**，存在以下主要問題：
1. 資料庫連接池配置過小
2. 缺少並發控制機制
3. 沒有頻率限制保護
4. 檔案上傳可能成為瓶頸
5. 缺少佇列機制

---

## 🔍 現有保護機制分析

### ✅ 已實作的功能

1. **資料庫事務保護**
   - 英檢報名使用 Sequelize 事務確保原子性
   - 預約系統使用行鎖定 (`lock: true`)

2. **唯一約束保護**
   - `studentId` 欄位有唯一約束，防止重複報名
   - 預約系統有複合唯一約束 `(eventId, studentId)`

3. **前端防重複提交**
   - 使用 `isSubmitting` 狀態防止重複點擊
   - 按鈕在提交過程中會被禁用

4. **錯誤處理**
   - 完整的錯誤處理機制
   - 捕獲並處理唯一約束違反錯誤

## ⚠️ 發現的主要問題

### 1. 資料庫連接池配置過小 ⚠️⚠️⚠️

**問題位置**：`reservation-backend/db.js`

```javascript
pool: {
  max: 5,    // ❌ 太小！只能同時處理 5 個資料庫連接
  min: 0,
  acquire: 30000,
  idle: 10000
}
```

**影響**：
- 當有超過 5 個學生同時報名時，後續請求會等待連接釋放
- 在高並發情況下，可能導致請求超時或連接池耗盡
- 文檔建議配置為 `max: 50, min: 10`，但實際配置遠低於此

**建議**：
```javascript
pool: {
  max: 50,          // 增加到 50
  min: 10,          // 保持最小連接數
  acquire: 30000,
  idle: 10000,
  evict: 1000,       // 添加清理間隔
  handleDisconnects: true
}
```

### 2. 英檢報名缺少行鎖定機制 ⚠️⚠️

**問題位置**：`reservation-backend/routes/englishTestRegistrationRouter.js:603-619`

```javascript
// 檢查是否已報名（使用事務確保原子性）
const transaction = await sequelize.transaction();

try {
  const existing = await EnglishTestRegistration.findOne({
    where: { studentId: formData.studentId },
    transaction  // ✅ 有事務
    // ❌ 但沒有 lock: true，可能導致競爭條件
  });
```

**對比預約系統**（有行鎖定）：
```javascript
const event = await Event.findByPk(eventId, { 
  lock: true,  // ✅ 使用行鎖定
  transaction
});
```

**影響**：
- 兩個學生同時使用相同學號報名時，可能都通過重複檢查
- 雖然有唯一約束保護，但會導致不必要的錯誤處理和回滾
- 增加資料庫負載和錯誤率

**建議**：
```javascript
const existing = await EnglishTestRegistration.findOne({
  where: { studentId: formData.studentId },
  lock: true,  // 添加行鎖定
  transaction
});
```

### 3. 缺少頻率限制（Rate Limiting）⚠️⚠️⚠️

**問題**：
- 文檔 `SYSTEM_PROTECTION_GUIDE.md` 提到有頻率限制機制
- 但實際代碼中**沒有找到 `express-rate-limit` 的實作**
- `server.js` 中沒有配置 rate limiting 中間件

**影響**：
- 單一學生可以快速發送大量請求
- 惡意用戶可以使用腳本進行攻擊
- 無法防止 DDoS 攻擊

**建議**：
```javascript
// server.js
const rateLimit = require('express-rate-limit');

// 報名請求限制：每分鐘最多 3 次
const registrationLimiter = rateLimit({
  windowMs: 60 * 1000, // 1 分鐘
  max: 3,
  message: '報名請求過於頻繁，請稍後再試',
  keyGenerator: (req) => {
    const studentId = req.body?.studentId || req.ip;
    return `registration:${req.ip}:${studentId}`;
  }
});

// 一般 API 限制：每分鐘最多 60 次
const generalLimiter = rateLimit({
  windowMs: 60 * 1000,
  max: 60
});

// 應用中間件
app.use('/api/english-test/register', registrationLimiter);
app.use('/api', generalLimiter);
```

### 4. 檔案上傳可能成為瓶頸 ⚠️⚠️

**問題位置**：`reservation-backend/routes/englishTestRegistrationRouter.js:99-115`

**問題**：
- 檔案上傳是同步處理，會阻塞請求
- 每個檔案最大 5MB，多個檔案同時上傳會消耗大量記憶體和 I/O
- 沒有檔案上傳的並發限制

**影響**：
- 大量學生同時上傳檔案會導致：
  - 記憶體使用激增
  - 磁碟 I/O 飽和
  - 請求處理時間延長
  - 可能導致伺服器崩潰

**建議**：
1. **使用佇列處理檔案上傳**
   ```javascript
   const Bull = require('bull');
   const fileUploadQueue = new Bull('file-upload', {
     redis: { host: 'localhost', port: 6379 }
   });
   ```

2. **限制同時上傳的檔案數量**
   ```javascript
   const uploadLimiter = rateLimit({
     windowMs: 60 * 1000,
     max: 2, // 每分鐘最多 2 次上傳
     message: '檔案上傳過於頻繁，請稍後再試'
   });
   ```

3. **使用非同步檔案處理**
   - 先保存報名資料
   - 檔案上傳在背景處理
   - 使用 CDN 或物件儲存服務（如 AWS S3）

### 5. 缺少請求佇列機制 ⚠️⚠️

**問題**：
- 所有報名請求都是同步處理
- 沒有優先級機制
- 沒有請求佇列來平滑處理高峰流量

**影響**：
- 當大量學生同時報名時，所有請求同時競爭資料庫資源
- 可能導致：
  - 資料庫連接池耗盡
  - 請求超時
  - 系統響應緩慢
  - 部分請求失敗

**建議**：
使用 **Bull** 或 **BullMQ** 實現請求佇列：

```javascript
const Bull = require('bull');
const registrationQueue = new Bull('registration', {
  redis: { host: 'localhost', port: 6379 },
  limiter: {
    max: 10,        // 每秒最多處理 10 個請求
    duration: 1000
  },
  defaultJobOptions: {
    attempts: 3,
    backoff: {
      type: 'exponential',
      delay: 2000
    }
  }
});

// 處理報名請求
registrationQueue.process(async (job) => {
  const { formData, files } = job.data;
  // 處理報名邏輯
  return await processRegistration(formData, files);
});

// API 端點
router.post('/english-test/register', async (req, res) => {
  const job = await registrationQueue.add({
    formData: req.body,
    files: req.files
  });
  
  res.json({ 
    message: '報名請求已加入佇列',
    jobId: job.id 
  });
});
```

### 6. 缺少並發控制鎖 ⚠️⚠️

**問題**：
- 沒有活動級別的並發控制
- 文檔提到「每活動最多 5 個併發請求」，但代碼中沒有實作

**建議**：
使用 Redis 實現分散式鎖：

```javascript
const Redis = require('ioredis');
const redis = new Redis();

async function acquireLock(key, ttl = 30) {
  const lockKey = `lock:${key}`;
  const result = await redis.set(lockKey, '1', 'EX', ttl, 'NX');
  return result === 'OK';
}

async function releaseLock(key) {
  const lockKey = `lock:${key}`;
  await redis.del(lockKey);
}

// 在報名處理中使用
router.post('/english-test/register', async (req, res) => {
  const lockKey = `registration:${req.body.studentId}`;
  
  if (!await acquireLock(lockKey)) {
    return res.status(429).json({ 
      error: '您的報名請求正在處理中，請勿重複提交' 
    });
  }
  
  try {
    // 處理報名邏輯
    const result = await processRegistration(req.body, req.files);
    return res.json(result);
  } finally {
    await releaseLock(lockKey);
  }
});
```
