# 班級統計資料為0的原因診斷報告

## 問題摘要

班級參與概況中所有統計資料都顯示為0：
- 至少參與人數: 0
- 參與率: 0.0%
- 簽到總次數: 0
- 平均參與次數: 0.00
- No-shows總數: 0

但班級名冊人數有正常顯示（如：39, 34, 40, 13, 39, 36），表示班級名單資料正常。

---

## 診斷結果

### 問題 1：學生ID匹配率低 ⚠️

**發現：**
- 班級學生ID總數：**200人**
- 預約學生ID總數：**667人**
- **匹配的學生ID數：僅58人**（29%）
- 班級中但沒有預約的學生：**142人**（71%）
- 有預約但不在班級中的學生：**609人**

**原因分析：**
- 班級名單中的學生（主要是B11、B12、B13、B14開頭）與實際預約的學生（主要是B08、B09、B10開頭）不匹配
- 這表示班級名單可能是新學期的學生，而預約資料是舊學期的學生

### 問題 2：所有預約都是「未簽到」狀態 ⚠️⚠️

**發現：**
- 所有2388筆預約記錄的 `checkinStatus` 都是「未簽到」
- 匹配的58個學生在114-1學期有預約，但**所有預約都是「未簽到」狀態**
- **沒有任何「已簽到」的記錄**

**範例：**
```
- B111010044: 總預約=2, 已簽到=0, 未簽到=2
- B113012038: 總預約=8, 已簽到=0, 未簽到=8
- B124030048: 總預約=19, 已簽到=0, 未簽到=19
```

**原因分析：**
- 統計查詢使用 `r.checkinStatus = '已簽到'` 來計算簽到數
- 由於所有記錄都是「未簽到」，所以統計結果為0

### 問題 3：114-1學期活動沒有預約 ⚠️

**發現：**
- 114-1學期範圍：2025-08-01 到 2026-01-31
- 114-1學期範圍內有93個活動
- 但這些活動（8月之後的活動）**都沒有預約記錄**
- 所有預約都在更早的日期（2025-04-14 到 2025-11-04）

**範例：**
```
- 2025-12-11 Job Talk: 預約數=0, 簽到數=0
- 2025-12-10 English Table: 預約數=0, 簽到數=0
- 2025-12-09 English Table: 預約數=0, 簽到數=0
```

---

## 根本原因

### 主要原因：簽到狀態問題

**統計查詢邏輯：**
```sql
WHERE r.checkinStatus = '已簽到'  -- 只計算已簽到的記錄
```

**實際資料狀態：**
- 所有預約記錄的 `checkinStatus` 都是「未簽到」
- 沒有任何「已簽到」的記錄

**影響：**
- `attendedCountTotal` = 0（因為沒有已簽到的記錄）
- `participatedCount` = 0（因為 `participatedStudents` Set 是空的）
- 所有統計指標都變成0

### 次要原因：學生ID不匹配

- 71%的班級學生沒有預約記錄
- 這會導致這些學生的統計為0（這是正常的，因為他們確實沒有參與）

---

## 解決方案

### 方案 1：檢查簽到功能是否正常運作（優先）

**問題：** 為什麼所有預約都是「未簽到」狀態？

**需要檢查：**
1. 簽到功能是否正常運作
2. 是否有進行過簽到操作
3. 簽到API是否正確更新 `checkinStatus` 欄位

**檢查步驟：**
```sql
-- 檢查是否有任何已簽到的記錄
SELECT checkinStatus, COUNT(*) as count
FROM Reservations
GROUP BY checkinStatus;

-- 檢查簽到時間欄位
SELECT 
  checkinStatus,
  checkinTime,
  COUNT(*) as count
FROM Reservations
WHERE checkinTime IS NOT NULL
GROUP BY checkinStatus, checkinTime IS NOT NULL;
```

### 方案 2：修改統計邏輯（如果簽到功能尚未啟用）

如果系統尚未開始使用簽到功能，可以考慮：

1. **暫時修改統計邏輯**：將「已簽到」改為「有預約就算參與」
2. **或者**：等待簽到功能正常運作後再查看統計

### 方案 3：更新班級名單

如果班級名單需要更新：
- 確認班級名單中的學生是否正確
- 確認是否需要匯入新學期的學生名單

## 建議的修復步驟

### 步驟 1：檢查簽到功能

```bash
# 檢查簽到相關的API是否正常
# 檢查是否有進行過簽到操作
```

### 步驟 2：測試簽到功能

1. 登入系統
2. 進入活動報表
3. 選擇一個有預約的活動
4. 嘗試進行簽到操作
5. 檢查資料庫中 `checkinStatus` 是否更新為「已簽到」

### 步驟 3：如果簽到功能正常，檢查資料

如果簽到功能正常，但資料仍然是「未簽到」，可能是：
- 歷史資料沒有簽到記錄（這是正常的）
- 需要對未來的活動進行簽到操作

### 步驟 4：考慮修改統計邏輯（可選）

如果希望「有預約就算參與」（不要求簽到），可以修改統計邏輯：

```javascript
// 修改前：只計算已簽到
WHERE r.checkinStatus = '已簽到'

// 修改後：計算所有預約（或已簽到+未簽到）
WHERE r.studentId IN (:studentIds)
  AND e.date BETWEEN :startDate AND :endDate
```

## 結論

**主要問題：** 所有預約記錄都是「未簽到」狀態，導致統計查詢結果為0。

**次要問題：** 71%的班級學生沒有預約記錄（這可能是正常的，如果他們確實沒有參與活動）。

**建議：** 
1. 優先檢查簽到功能是否正常運作
2. 確認是否有進行過簽到操作
3. 如果簽到功能尚未啟用，考慮修改統計邏輯或等待功能啟用
