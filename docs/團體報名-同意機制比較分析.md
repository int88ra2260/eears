# 團體報名同意機制比較分析

## 一、方案概述

團體報名需要所有成員在 24 小時內完成同意，目前考慮兩種方式：
1. **驗證碼（Verification Code）**
2. **電郵授權（Email Authorization Link）**

---

## 二、方式一：驗證碼（Verification Code）

### 2.1 運作流程

```
1. 團體報名送出成功
   ↓
2. 系統產生唯一驗證碼（例如：ABC123）
   ↓
3. 發送郵件給成員，郵件包含：
   - 團體報名資訊
   - 驗證碼：ABC123
   - 驗證碼輸入頁面連結
   ↓
4. 成員訪問驗證碼輸入頁面
   ↓
5. 成員輸入驗證碼
   ↓
6. 系統驗證驗證碼是否正確且未使用
   ↓
7. 驗證成功 → 標記該成員為「已同意」
```

### 2.2 優點

| 優點 | 說明 |
|------|------|
| **安全性較高** | 驗證碼通常有時效性（24 小時），且為一次性使用，降低被濫用風險 |
| **可追蹤使用記錄** | 系統可記錄驗證碼的使用時間、IP、User-Agent，便於稽核與問題排查 |
| **不依賴郵件連結** | 即使郵件被轉發，沒有驗證碼也無法完成同意 |
| **使用者體驗可控** | 成員需主動訪問系統頁面輸入驗證碼，可確保成員知道同意流程與後續步驟 |
| **可加入額外驗證** | 可在輸入驗證碼時要求登入或輸入身分證字號，進一步確認身份 |

### 2.3 缺點

| 缺點 | 說明 |
|------|------|
| **操作步驟較多** | 成員需先查看郵件取得驗證碼，再訪問系統頁面輸入，步驟較繁瑣 |
| **可能遺失驗證碼** | 若成員誤刪郵件或忘記驗證碼，需重新發送（需提供「重新發送驗證碼」功能） |
| **需要額外頁面** | 需開發「輸入驗證碼」的頁面與 API，增加開發與維護成本 |
| **可能被誤用** | 若驗證碼被他人看到（例如：郵件被轉發、螢幕截圖），可能被惡意使用（但可透過 IP 或登入狀態限制） |
| **增加系統負擔** | 需儲存驗證碼、驗證碼使用狀態，增加資料庫查詢與驗證邏輯 |

### 2.4 實作複雜度

**開發項目：**
- ✅ 驗證碼產生邏輯（6 位數字或字母數字組合，確保唯一性）
- ✅ 驗證碼儲存（資料庫欄位：`approvalCode`）
- ✅ 驗證碼輸入頁面（前端：`/group/approve/:code` 或獨立頁面）
- ✅ 驗證碼驗證 API（`POST /api/group/approve`，驗證 code 是否正確且未使用）
- ✅ 驗證碼重新發送功能（`POST /api/group/resend-code`）
- ✅ 驗證碼使用記錄（記錄使用時間、IP、User-Agent）

**複雜度評分：** ⭐⭐⭐（中等）

## 三、方式二：電郵授權（Email Authorization Link）

### 3.1 運作流程

```
1. 團體報名送出成功
   ↓
2. 系統產生唯一授權 token（例如：UUID 或 JWT）
   ↓
3. 發送郵件給成員，郵件包含：
   - 團體報名資訊
   - 授權連結：https://example.com/group/approve?token=xxx
   ↓
4. 成員點擊授權連結
   ↓
5. 系統驗證 token 是否正確、未使用、未過期
   ↓
6. 驗證成功 → 標記該成員為「已同意」，token 標記為已使用
   ↓
7. 顯示成功頁面
```

### 3.2 優點

| 優點 | 說明 |
|------|------|
| **操作簡單快速** | 成員只需點擊郵件中的連結即可完成同意，無需額外輸入，操作步驟最少 |
| **使用者體驗佳** | 一鍵完成，降低操作門檻，提高完成率 |
| **實作相對簡單** | 只需產生授權 token 並嵌入連結，點擊後驗證 token 即可，開發與維護成本較低 |
| **自動導向** | 點擊後可直接導向成功頁面，流程順暢 |
| **符合常見做法** | 多數系統（如報名確認、訂閱確認、密碼重設）採用此方式，使用者熟悉 |

### 3.3 缺點

| 缺點 | 說明 |
|------|------|
| **安全性較低** | 若郵件被轉發或洩漏，連結可能被他人點擊（可透過 token 時效性與一次性使用降低風險） |
| **郵件連結可能失效** | 若郵件系統過濾連結或標記為垃圾郵件，可能影響使用 |
| **難以追蹤操作者** | 僅能記錄點擊時間與 IP，無法確認是否為本人操作（但通常可透過 Email 擁有權假設） |
| **可能被誤點** | 若郵件被誤點或自動預覽，可能造成非預期的同意（可透過二次確認降低風險） |
| **Token 管理** | 需管理 token 的產生、儲存、驗證、失效，但邏輯相對簡單 |

### 3.4 實作複雜度

**開發項目：**
- ✅ Token 產生邏輯（UUID 或 JWT，確保唯一性）
- ✅ Token 儲存（資料庫欄位：`approvalToken`）
- ✅ 授權連結驗證 API（`GET /api/group/approve/:token`，驗證 token 是否正確、未使用、未過期）
- ✅ Token 使用記錄（記錄使用時間、IP、User-Agent）
- ✅ Token 重新發送功能（`POST /api/group/resend-link`，產生新 token 並發送新連結）

**複雜度評分：** ⭐⭐（較低）

## 四、安全性比較

| 項目 | 驗證碼 | 電郵授權 |
|------|--------|----------|
| **一次性使用** | ✅ 支援 | ✅ 支援 |
| **時效性** | ✅ 24 小時 | ✅ 24 小時 |
| **防轉發** | ✅ 較佳（需驗證碼） | ⚠️ 較弱（但可透過 token 唯一性降低風險） |
| **操作追蹤** | ✅ 可記錄輸入時間、IP | ✅ 可記錄點擊時間、IP |
| **身份確認** | ✅ 可要求登入或額外驗證 | ⚠️ 僅能假設 Email 擁有權 |
| **誤用風險** | ⚠️ 中等（若驗證碼洩漏） | ⚠️ 中等（若連結洩漏） |

**結論：** 驗證碼安全性略高，但電郵授權的安全性已足夠（透過 token 時效性、一次性使用、IP 記錄等方式）。

---

## 五、使用者體驗比較

| 項目 | 驗證碼 | 電郵授權 |
|------|--------|----------|
| **操作步驟** | 2 步（查看郵件 → 輸入驗證碼） | 1 步（點擊連結） |
| **操作時間** | 約 30 秒～1 分鐘 | 約 5 秒 |
| **學習成本** | 需了解如何輸入驗證碼 | 無需學習，點擊即可 |
| **錯誤率** | 可能輸入錯誤（需重新輸入） | 幾乎無錯誤（點擊即可） |
| **完成率預估** | 較低（步驟多，可能遺忘） | 較高（步驟少，操作簡單） |

**結論：** 電郵授權的使用者體驗明顯優於驗證碼。

---

## 六、實作成本比較

| 項目 | 驗證碼 | 電郵授權 |
|------|--------|----------|
| **前端開發** | 需開發驗證碼輸入頁面 | 僅需授權連結處理頁面（較簡單） |
| **後端開發** | 需驗證碼產生、驗證、重新發送 API | 需 token 產生、驗證、重新發送 API（較簡單） |
| **資料庫設計** | 需 `approvalCode` 欄位 | 需 `approvalToken` 欄位 |
| **維護成本** | 需維護驗證碼輸入頁面與邏輯 | 僅需維護 token 驗證邏輯（較簡單） |
| **測試複雜度** | 需測試驗證碼輸入、錯誤處理、重新發送 | 需測試 token 驗證、錯誤處理、重新發送（較簡單） |

**結論：** 電郵授權的實作與維護成本較低。

## 七、建議方案

### 7.1 推薦方案：電郵授權（Email Authorization Link）

**理由：**
1. ✅ **使用者體驗優先**：團體報名需要所有成員配合，降低操作門檻有助於提高完成率
2. ✅ **實作簡單快速**：開發與維護成本較低，可快速上線
3. ✅ **安全性可接受**：透過 token 時效性（24 小時）、一次性使用、IP 記錄等方式，安全性已足夠
4. ✅ **符合常見做法**：多數系統採用此方式，使用者熟悉且信任

### 7.2 安全強化措施

為確保電郵授權的安全性，建議實作以下措施：

| 措施 | 說明 | 實作方式 |
|------|------|----------|
| **Token 時效性** | Token 僅在 24 小時內有效 | 檢查 `createdAt + 24小時` 是否超過當前時間 |
| **一次性使用** | Token 使用後即失效 | 標記 `approvalStatus = 'approved'`，再次使用時拒絕 |
| **IP 記錄** | 記錄點擊時的 IP | 記錄在 `TeamMember` 表或日誌中 |
| **User-Agent 記錄** | 記錄點擊時的瀏覽器資訊 | 記錄在 `TeamMember` 表或日誌中 |
| **Token 唯一性** | 確保每個成員的 token 唯一 | 使用 UUID 或 JWT，並檢查資料庫唯一性 |
| **可選：二次確認** | 點擊後顯示確認頁面 | 顯示「您確定要同意此團體報名嗎？」確認按鈕 |

### 7.3 若需更高安全性

若未來需要更高的安全性，可考慮：

1. **雙重驗證**：電郵授權 + 簡訊驗證碼（但會增加複雜度與成本）
2. **登入要求**：點擊連結後要求成員登入系統（但會降低使用者體驗）
3. **提供兩種方式**：讓成員選擇驗證碼或電郵授權（但會增加開發與維護成本）

## 八、實作建議

### 8.1 資料庫欄位設計

```sql
-- TeamMember 表
approvalToken VARCHAR(255) UNIQUE,  -- 授權 token（UUID）
approvalTokenExpiresAt DATETIME,     -- Token 過期時間（createdAt + 24小時）
approvalStatus ENUM('pending', 'approved', 'expired'),  -- 同意狀態
approvedAt DATETIME,                 -- 同意時間
approvalIp VARCHAR(45),              -- 同意時的 IP（IPv4 或 IPv6）
approvalUserAgent TEXT,               -- 同意時的 User-Agent
```

### 8.2 API 設計

```
POST /api/group/registrations
  - 建立團體報名，產生 token，發送郵件

GET /api/group/approve/:token
  - 驗證 token，完成同意，顯示成功頁面

POST /api/group/resend-link/:teamId/:memberId
  - 重新產生 token 並發送新連結（僅在 pending 狀態下）
```

### 8.3 郵件模板

```
主旨：[培力英檢] 團體報名確認通知

親愛的 {name} 您好，

您已被加入培力英檢團體報名，團體資訊如下：

團體編號：{teamId}
團體成員：
{memberList}

請在 24 小時內點擊以下連結完成同意：
{approvalLink}

若您未點擊連結，此團體報名將在 24 小時後自動失效。

如有疑問，請聯繫相關單位。
```
