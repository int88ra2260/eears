# 培力英檢團體報名設計構想

## 一、基本架構

### 1.1 獨立路徑（路由）
- **獨立網址**：團體報名需有獨立的路徑（網址），與個人報名分離
- **建議路由**：`/register/english-test/group` 或 `/register/english-test/learning-partner`
- **與個人報名關係**：個人報名與團體報名為兩個獨立的報名流程

---

## 二、報名流程

### 2.1 選擇報名人數
- **第一步**：學生進入團體報名頁面後，需先選擇報名人數
- **選項**：3 人、4 人（符合專案辦法：每團 3～4 人）
- **動態表單**：根據選擇的報名人數，動態顯示對應數量的報名欄位組

### 2.2 填寫成員資料
- **欄位組**：每組包含「學號」與「姓名」兩個欄位
- **顯示邏輯**：
  - 選擇 2 人 → 顯示 2 組欄位（學號 + 姓名）
  - 選擇 3 人 → 顯示 3 組欄位（學號 + 姓名）
  - 選擇 4 人 → 顯示 4 組欄位（學號 + 姓名）
- **驗證**：每組欄位需填寫完整（學號與姓名皆必填）

---

## 三、報名資格檢查

### 3.1 檢查所有成員的報名狀態
- **檢查時機**：當學生填寫完所有成員資料並點擊「送出」時
- **檢查邏輯**：系統需檢查填入的**所有學生資料**是否已經是「報名成功」的狀態
  - 查詢條件：學號 + 姓名（或加上身分證字號以確保唯一性）
  - 查詢目標：`EnglishTestRegistration` 資料表中的個人報名記錄
  - 狀態判斷：該筆個人報名記錄的狀態需為 `status = 'success'`（報名成功）
- **送出條件**：**必須所有人都是「報名成功」的狀態**（`status = 'success'`），才能送出該筆團體報名
- **代表者檢查**：代表者（填寫表單的學生）也必須是「報名成功」狀態（`status = 'success'`）
- **錯誤處理**：若有任何一位成員（包含代表者）未報名成功，顯示錯誤訊息並列出不符合資格的成員

---

## 四、自動通知機制

### 4.1 發送通知郵件
- **觸發時機**：團體報名送出成功後
- **收件人**：該組報名的**所有學生**（從個人報名記錄中取得 Email）
- **郵件內容**：告知團體報名資訊，包含：
  - 團體報名編號/ID
  - 團體成員名單（所有成員的學號與姓名）
  - 報名時間
  - 後續步驟說明（需在 24 小時內同意）
  - 同意連結或驗證碼

---

## 五、成員同意機制

### 5.1 同意方式（二選一）
- **方式一：驗證碼**
  - 郵件中包含一組驗證碼
  - 成員需登入系統或訪問特定頁面，輸入驗證碼完成同意
- **方式二：電郵授權**
  - 郵件中包含「同意」連結（帶有授權 token）
  - 成員點擊連結後自動完成同意（無需登入）

### 5.2 同意時限
- **時限**：**24 小時內**
- **起算時間**：團體報名送出成功的時間
- **完成條件**：**所有成員**都需在 24 小時內完成同意
- **狀態追蹤**：系統需記錄每位成員的同意狀態與時間

---

## 六、自動失效機制

### 6.1 失效條件
- **觸發時機**：團體報名送出後 24 小時
- **失效條件**：若在 24 小時內，**有任何一位成員未完成同意**，該組團體報名自動失效
- **失效動作**：
  - 將團體報名記錄狀態設為「已失效」或「已取消」
  - **釋放該團體佔用的名額**（名額在「送出」時立即佔用，失效後釋出）
  - 可選：發送通知給已同意的成員，告知團體報名已失效
- **重新報名**：失效後允許重新報名，**無需冷卻期**

### 6.2 失效檢查
- **檢查方式**：定時任務（cron job）或背景任務，每小時或每 15 分鐘檢查一次
- **檢查邏輯**：
  - 查詢所有「待同意」狀態的團體報名
  - 計算送出時間與當前時間的差距
  - 若超過 24 小時且仍有成員未同意，標記為失效

---

## 七、資料結構建議

### 7.1 團體報名記錄（LearningPartnerTeam）
- `id`：團體報名 ID
- `teamName`：團體名稱（選填，或自動產生）
- `representativeStudentId`：代表者學號（填寫表單的學生）
- `teamSize`：報名人數（2/3/4）
- `status`：狀態（`pending_approval` 待同意、`approved` 已同意、`expired` 已失效、`cancelled` 已取消）
- `createdAt`：送出時間（用於計算 24 小時時限）
- `expiresAt`：過期時間（`createdAt + 24小時`）
- `approvedAt`：全員同意完成的時間

### 7.2 團體成員記錄（TeamMember）
- `id`：成員記錄 ID
- `teamId`：團體報名 ID（外鍵）
- `studentId`：學號
- `name`：姓名
- `email`：Email（從個人報名記錄取得）
- `isRepresentative`：是否為代表（填表人）
- `approvalStatus`：同意狀態（`pending` 待同意、`approved` 已同意、`expired` 已過期）
- `approvalCode`：驗證碼（若使用驗證碼方式）
- `approvalToken`：授權 token（若使用電郵授權方式）
- `approvedAt`：同意時間
- `personalRegistrationId`：對應的個人報名記錄 ID（外鍵，關聯到 `EnglishTestRegistration`）

---

## 八、系統流程圖

```
1. 學生進入團體報名頁面（獨立路徑）
   ↓
2. 選擇報名人數（2/3/4 人）
   ↓
3. 填寫對應數量的成員資料（學號 + 姓名）
   ↓
4. 點擊「送出」
   ↓
5. 系統檢查所有成員是否已「報名成功」
   ├─ 否 → 顯示錯誤，列出不符合資格的成員
   └─ 是 → 建立團體報名記錄（狀態：待同意）
   ↓
6. 系統自動發送郵件給所有成員
   ↓
7. 成員在 24 小時內完成同意（驗證碼或電郵授權）
   ├─ 24 小時內全員同意 → 團體報名狀態改為「已同意」
   └─ 24 小時內有成員未同意 → 定時任務標記為「已失效」
```

## 九、與現有系統的整合

### 9.1 與個人報名的關聯
- **前置條件**：所有成員（包含代表者）必須先完成個人培力英檢報名（狀態為 `status = 'success'`）
- **資料關聯**：`TeamMember.personalRegistrationId` 關聯到 `EnglishTestRegistration.id`
- **檢查 API**：需提供 API 檢查學號 + 姓名是否對應到 `status = 'success'` 的個人報名記錄

### 9.2 與專案辦法的對應
- **每團 3～4 人**：由選擇報名人數限制
- **每人限參與一組**：在檢查報名狀態時，同時檢查該學號是否已在其他「已同意」或「待同意」狀態的團體中
- **50 組為限**：名額在「送出」時立即佔用，僅計算「已同意」和「待同意」狀態的團體，滿 50 組即不再開放團體報名
- **名額釋放**：失效的團體會釋出名額，其他學生可立即使用該名額重新報名

## 十、確認事項與決策

### 10.1 已確認事項

1. **「報名成功」的定義**：僅有 `status = 'success'`（報名成功）狀態
2. **代表者身份**：代表者也需要是報名狀態為 `success` 的學生
3. **失效後的處理**：允許重新報名，無須冷卻期
4. **名額計算時機**：「送出」後立即佔用名額，等待失效後釋出名額

### 10.2 同意機制比較分析

#### 方式一：驗證碼（Verification Code）

**優點：**
- ✅ **安全性較高**：驗證碼通常有時效性（例如 24 小時），且為一次性使用
- ✅ **可追蹤使用記錄**：系統可記錄驗證碼的使用時間與 IP，便於稽核
- ✅ **不依賴郵件連結**：即使郵件被轉發，沒有驗證碼也無法完成同意
- ✅ **使用者體驗可控**：成員需主動訪問系統頁面輸入驗證碼，可確保成員知道同意流程

**缺點：**
- ❌ **操作步驟較多**：成員需先查看郵件取得驗證碼，再訪問系統頁面輸入，步驟較繁瑣
- ❌ **可能遺失驗證碼**：若成員誤刪郵件或忘記驗證碼，需重新發送
- ❌ **需要額外頁面**：需開發「輸入驗證碼」的頁面與 API
- ❌ **可能被誤用**：若驗證碼被他人看到，可能被惡意使用（但可透過 IP 或登入狀態限制）

**實作複雜度：** 中等
- 需產生唯一驗證碼（例如：6 位數字或字母數字組合）
- 需建立驗證碼輸入頁面
- 需驗證碼驗證 API
- 需記錄驗證碼使用狀態

---

#### 方式二：電郵授權（Email Authorization Link）

**優點：**
- ✅ **操作簡單快速**：成員只需點擊郵件中的連結即可完成同意，無需額外輸入
- ✅ **使用者體驗佳**：一鍵完成，降低操作門檻
- ✅ **實作相對簡單**：只需產生授權 token 並嵌入連結，點擊後驗證 token 即可
- ✅ **自動導向**：點擊後可直接導向成功頁面，流程順暢

**缺點：**
- ❌ **安全性較低**：若郵件被轉發或洩漏，連結可能被他人點擊（可透過 token 時效性與一次性使用降低風險）
- ❌ **郵件連結可能失效**：若郵件系統過濾連結或標記為垃圾郵件，可能影響使用
- ❌ **難以追蹤操作者**：僅能記錄點擊時間與 IP，無法確認是否為本人操作（但通常可透過 Email 擁有權假設）
- ❌ **可能被誤點**：若郵件被誤點或自動預覽，可能造成非預期的同意

**實作複雜度：** 較低
- 需產生唯一授權 token（例如：UUID 或 JWT）
- 需建立授權連結驗證 API
- 需記錄 token 使用狀態（一次性使用）
- Token 需有時效性（24 小時）

---

#### 建議方案

**推薦：電郵授權（Email Authorization Link）**

**理由：**
1. **使用者體驗優先**：團體報名需要所有成員配合，降低操作門檻有助於提高完成率
2. **實作簡單快速**：開發與維護成本較低
3. **安全性可接受**：透過 token 時效性（24 小時）、一次性使用、IP 記錄等方式，安全性已足夠
4. **符合常見做法**：多數系統（如報名確認、訂閱確認）採用此方式

**安全強化措施：**
- Token 設定 24 小時時效
- Token 一次性使用（使用後即失效）
- 記錄點擊時間、IP、User-Agent
- 可選：加入 CSRF token 或 referer 檢查

**若需更高安全性：**
- 可採用「電郵授權 + 簡訊驗證碼」的雙重驗證（但會增加複雜度與成本）
- 或提供兩種方式讓成員選擇（但會增加開發與維護成本）

## 十一、更新後的系統流程圖

```
1. 學生進入團體報名頁面（獨立路徑：/register/english-test/group）
   ↓
2. 選擇報名人數（2/3/4 人）
   ↓
3. 填寫對應數量的成員資料（學號 + 姓名）
   ↓

   ↓

   └─ 是 → 檢查名額是否已滿（已佔用 < 50）
       ├─ 已滿 → 顯示「名額已滿」
       └─ 未滿 → 建立團體報名記錄（狀態：pending_approval，立即佔用名額）
   ↓

   ├─ 24 小時內全員同意 → 團體報名狀態改為「已同意」（approved）
   └─ 24 小時內有成員未同意 → 定時任務標記為「已失效」（expired），釋出名額
   ↓
8. 失效後可立即重新報名（無冷卻期）
```
